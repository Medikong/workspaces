# aws-dev 지속 E2E 검증 아키텍처

aws-dev 지속 E2E 검증은 배포된 서비스가 장시간 동안 계속 정상 동작하는지 확인하기 위한 synthetic test다.

목표는 부하를 만드는 것이 아니라, 실제 사용자가 거치는 진입 경로와 핵심 예매 흐름이 계속 살아 있는지 반복해서 확인하는 것이다.

관련 이슈:

- 상위 이슈: Medikong/workspace#29
- GitOps 실행 범위: Medikong/gitops#23

## 목적

현재 aws-dev에는 인프라와 서비스가 함께 배포되어 있다. 단순히 Pod가 Running인지 보는 것만으로는 실제 서비스 흐름이 살아 있는지 알기 어렵다.

지속 E2E 검증은 다음 질문에 답하기 위한 장치다.

```text
Ingress로 들어온 요청이 서비스까지 도달하는가?
공연, 좌석, 예약, 결제, 티켓 흐름이 한 사이클로 이어지는가?
DB, Kafka 같은 런타임 의존성 문제가 실제 요청에서 드러나는가?
24시간 반복 실행 중 특정 시간대에 실패가 몰리는가?
테스트 데이터가 계속 쌓여 운영 판단을 방해하지 않는가?
```

## 검증 범위

1차 검증은 핵심 예매 흐름 한 사이클에 집중한다.

```text
공연 목록 조회
-> 좌석 조회
-> 예약 생성
-> 결제 요청
-> 티켓 발급 확인
-> 최종 상태 조회
```

처음부터 모든 예외 흐름을 포함하지 않는다. 목적은 aws-dev의 지속 동작 여부를 빠르게 확인하는 것이므로, 정상 흐름 하나를 안정적으로 반복하는 것이 먼저다.

## 실행 위치

실행 위치는 Kubernetes 안의 별도 네임스페이스로 둔다.

```text
synthetic namespace
-> synthetic client 실행 단위
-> Ingress URL 호출
-> 실행 결과 기록
```

별도 네임스페이스를 두는 이유는 애플리케이션 서비스와 실행 범위를 분리하기 위해서다. 서비스 네임스페이스에 테스트 실행 리소스가 섞이면 배포 상태를 볼 때 운영 리소스와 검증 리소스가 구분되지 않는다.

완전한 외부 독립성을 목표로 하지는 않는다. 지금 단계의 목표는 빠르게 시도할 수 있고 GitOps로 관리하기 쉬운 구조를 만드는 것이다.

## 요청 경로

테스트 클라이언트는 내부 Service DNS가 아니라 Ingress 주소를 기준으로 요청한다.

```text
synthetic client
-> aws-dev Ingress
-> service
-> database / kafka
```

이 경로는 실제 사용자가 들어오는 경로에 더 가깝다. Ingress, 라우팅, TLS, 서비스 연결, 런타임 의존성까지 함께 확인할 수 있다.

단, 클러스터 내부에서 Ingress 주소를 다시 호출하는 구조이므로 aws-dev 네트워크에서 이 경로가 실제로 가능한지 첫 실행에서 확인해야 한다.

## 실행 주기

초기 주기는 10분을 기본 후보로 둔다. 흐름이 안정적으로 확인되면 5분으로 줄일 수 있다.

실행은 짧게 끝나는 1회 시나리오여야 한다. 이전 실행이 끝나지 않았는데 다음 실행이 겹치면 검증 결과가 흐려지고, 테스트 데이터도 예상보다 많이 쌓일 수 있다.

운영 판단 기준은 한 번의 실패보다 반복 실패에 둔다.

```text
1회 실패
-> 일시 오류 가능성 확인

연속 실패
-> 서비스 흐름 또는 인프라 경로 이상 후보

24시간 실패 분포
-> 특정 시간대, 배포 직후, 리소스 압박 구간 확인
```

## 테스트 데이터

반복 실행에서 가장 먼저 관리해야 할 것은 테스트 데이터다.

가능한 방향은 세 가지다.

```text
고정 synthetic 공연과 좌석 사용
매 실행마다 데이터 생성 후 정리
조회 중심의 가벼운 흐름부터 시작
```

1차 방향은 고정 synthetic 데이터를 사용하는 것이다. 매 실행마다 새로운 업무 데이터를 계속 만들면 DB가 빠르게 오염되고, 24시간 결과를 볼 때 실제 데이터와 테스트 데이터가 섞인다.

고정 데이터는 다음 기준을 만족해야 한다.

- 실제 고객 흐름과 구분되는 식별자를 가진다.
- 반복 실행 후에도 다음 실행이 같은 조건으로 시작할 수 있다.
- 실패 시 남은 중간 상태를 확인할 수 있다.
- 운영 대시보드나 보고에서 synthetic 데이터임을 구분할 수 있다.

## 결과 확인

초기 결과 확인은 실행 로그로 충분하다.

```text
실행 시각
대상 base URL
성공 여부
실패 단계
응답 코드
응답 시간
요청 식별자
```

이후에는 Loki, Prometheus, Grafana, Alertmanager와 연결할 수 있다.

```text
1단계: 실행 로그에서 성공/실패 확인
2단계: Loki에서 실행 이력 조회
3단계: Prometheus/Grafana에서 성공률과 응답시간 확인
4단계: Alertmanager로 연속 실패 알림
```

처음부터 대시보드와 알림까지 모두 묶지 않는다. 먼저 aws-dev에서 반복 실행이 안정적으로 되는지 확인한 뒤, 결과 수집과 알림을 붙인다.

## Repo별 책임

```text
workspace
-> 목적, 검증 범위, 성공 기준, 운영 판단 기준 문서화

service
-> synthetic client 이미지, Newman collection, 테스트 시나리오

gitops
-> aws-dev 실행 범위, 네임스페이스, 설정, 실행 주기, 리소스 제한

infra
-> 별도 실행 노드나 네트워크 기반이 필요해질 때만 관여
```

현재 1차 범위는 `workspace`, `service`, `gitops`에 걸친다. 별도 EC2 실행기나 외부 synthetic monitoring이 필요해지기 전까지 `infra` 작업은 만들지 않는다.

## 완료 기준

1차 설계의 완료 기준은 다음과 같다.

- aws-dev에서 핵심 예매 흐름이 반복 실행된다.
- 요청은 Ingress 주소를 기준으로 들어간다.
- 실행 결과가 성공/실패로 남는다.
- 실패한 단계와 응답 상태를 확인할 수 있다.
- 테스트 데이터가 반복 실행을 방해하지 않는다.
- 24시간 결과를 보고 지속 동작 여부를 판단할 수 있다.
