공연 티켓 발행 서비스 구현 설계

1. 서비스 목표와 성공 기준

이 프로젝트는 인기 공연 티켓 오픈 상황을 가정한다. 짧은 시간에 많은 사용자가 접속하고, 같은 좌석을 동시에 선택하며, 예약, 결제, 티켓 발행, 알림, 티켓 파일 저장이 연쇄적으로 발생한다.

목표는 단순 CRUD가 아니라 동시성 제어, 이벤트 기반 처리, 장애 격리, 관측성, 서비스 메시, 클라우드 저장소를 실제 문제 해결 기준으로 적용하고, 적용 전후 차이를 가능한 한 수치로 검증하는 것이다.

프로젝트 목표 성공 기준 측정 지표
좌석 중복 발행 방지 동시 예매 요청에서도 같은 좌석의 유효 티켓은 1장만 생성된다. duplicate_ticket_count = 0
핵심 API 응답성과 후속 처리 분리 예약 생성 API는 결제/티켓/알림 후속 처리와 분리된다. reservation_api_p95_ms, ticket_issue_delay_ms
장애 격리 알림 장애가 티켓 발행 성공 흐름을 실패시키지 않는다. core_flow_success_rate, notification_retry_count
운영 가시성 확보 장애 상황에서 metric, log, trace로 병목 서비스를 찾을 수 있다. service_p95_latency_ms, service_5xx_rate, trace_duration_ms
서비스 간 통신 보안과 트래픽 제어 mTLS와 traffic split으로 내부 통신 보안과 점진 배포를 검증한다. mtls_enabled_services, traffic_split_ratio
티켓 artifact 저장소 분리 티켓 QR/PDF는 pod 내부가 아니라 S3에 저장된다. ticket_artifact_upload_success_rate, s3_object_count

2. 추천 서비스 구성

서비스 책임 저장소 핵심 API
auth-service 로그인, JWT 발급, 역할 관리 PostgreSQL POST /auth/login, POST /auth/refresh
concert-service 공연, 공연장, 회차, 좌석 배치 조회 PostgreSQL GET /concerts, GET /concerts/{id}, GET /concerts/{id}/seats
reservation-service 좌석 임시 점유, 예약 생성, 예약 만료 처리 PostgreSQL, Redis 후보 POST /reservations, GET /reservations/{id}, POST /reservations/{id}/cancel
payment-service 결제 mock, 결제 승인/실패 이벤트 발행 PostgreSQL POST /payments, GET /payments/{id}
ticket-service 티켓 발행, QR/PDF 생성, S3 저장, 티켓 조회 PostgreSQL, S3 POST /tickets/issue, GET /tickets/{id}, GET /tickets/me
notification-service 예약/결제/티켓 이벤트 기반 알림 저장 및 발송 mock MongoDB GET /notifications, GET /notifications/{id}
dashboard 사용자 예매 화면과 운영 데모 화면 S3 static hosting 또는 container 정적 HTML/JS
결정: Kafka는 도메인 이벤트 스트림에 사용한다. RabbitMQ는 작업 큐에 적합하지만, 이 프로젝트에서는 consumer lag, replay, 여러 consumer group, 이벤트 기반 장애 격리를 보여주기 위해 Kafka를 선택한다.

3. 전체 흐름

사용자
  -> Kong Gateway
  -> auth-service 로그인
  -> concert-service 공연/좌석 조회
  -> reservation-service 좌석 임시 점유 및 예약 생성
  -> payment-service 결제 mock
  -> Kafka payment-approved
  -> ticket-service 티켓 발행 및 S3 저장
  -> Kafka ticket-issued
  -> notification-service 알림 저장
  -> dashboard에서 티켓/알림 확인

6. 좌석 중복 방지 구현 전략

티켓 서비스의 핵심 invariant는 한 공연 회차의 한 좌석에는 하나의 유효 예약 또는 티켓만 존재한다이다.

1차 구현

  • reservation-service가 좌석 점유의 단일 책임자가 된다.
  • seat_locks 또는 reservations에 concertId + seatId unique constraint를 둔다.
  • 동시 요청 중 하나만 성공하고 나머지는 409 Conflict를 반환한다.
  • 결제 시간 제한이 지나면 예약을 만료하고 좌석을 다시 열 수 있다.

심화 후보

  • Redis distributed lock
  • 예약 만료 scheduler
  • Kafka compacted topic 기반 좌석 상태 projection
