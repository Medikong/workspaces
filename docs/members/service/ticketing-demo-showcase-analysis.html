공연 티켓 서비스 목표와 수치화 전략

1. 프로젝트 목표 설정 방향

이 프로젝트의 목표는 기능 목록을 많이 만드는 것이 아니다. 티켓 오픈 상황에서 발생하는 문제를 구조적으로 줄이고, 그 차이를 가능한 한 수치로 검증하는 것이다.

핵심 메시지: 이 구조는 트래픽 폭주, 좌석 중복, 장애 전파, 운영 가시성 부족 문제를 줄이기 위해 설계되었고, 각 목표는 측정 지표와 검증 시나리오를 가진다.

2. 목표와 성공 지표

문제 구조적 해결 측정 지표 성공 기준
같은 좌석 중복 예매 reservation-service가 좌석 점유 단일 책임, DB unique constraint 또는 lock duplicate_ticket_count, reservation_conflict_count 부하 테스트 후 중복 티켓 수가 0이어야 한다.
후속 처리 때문에 API 응답 지연 Kafka 이벤트 기반 티켓 발행/알림 분리 reservation_api_p95_ms, ticket_issue_delay_ms 후속 처리를 API 응답 경로에서 분리하고 p95 latency를 비교한다.
알림 장애가 전체 예매 실패로 전파 core flow와 notification flow 분리 core_flow_success_rate, notification_retry_count notification 장애 중에도 결제 완료 티켓은 발행되어야 한다.
서비스 간 통신 보안 부족 Istio mTLS mtls_enabled_services, service_to_service_rps mesh 대상 서비스 간 통신이 mTLS로 보호됨을 검증한다.
배포 중 위험 Istio traffic split 또는 canary 후보 traffic_split_ratio, mesh_5xx_rate 설정한 traffic split 비율이 실제 요청 분포에 반영된다.
장애 원인 파악 어려움 Prometheus, Grafana, Loki, Tempo p95_latency, 5xx_rate, consumer_lag, trace_duration 대시보드와 trace로 병목 서비스를 찾는다.
티켓 파일 저장 안정성 부족 S3 object storage ticket_artifact_upload_success_rate, s3_object_count 티켓 QR/PDF가 S3에 저장되고 DB에는 key만 남아야 한다.

3. 핵심 검증 시나리오 4개

검증 1. 좌석 중복 방지

같은 공연, 같은 좌석에 동시 요청을 넣는다. 성공은 1건만 발생하고 나머지는 409 Conflict가 되어야 한다.

목표 지표: duplicate_ticket_count = 0

검증 2. Kafka 기반 후속 처리 분리

결제 승인 이후 payment-approved 이벤트가 발행되고, ticket-service와 notification-service가 독립적으로 처리한다.

목표 지표: 예약 API p95 latency와 티켓 발행 지연을 분리해서 측정한다.

검증 3. 장애 격리

notification-service를 중단한 상태에서 결제와 티켓 발행이 유지되는지 검증한다.

목표 지표: core_flow_success_rate는 유지되고 notification_retry_count만 증가한다.

검증 4. Service Mesh와 Observability

Istio mTLS, traffic split, Grafana dashboard, trace를 함께 검증한다.

목표 지표: mtls_enabled_services, traffic_split_ratio, service_p95_latency_ms

7. 결론

공연 티켓 서비스는 심화 프로젝트를 보여주기에 적합하다. 좌석 동시성, 트래픽 폭주, 이벤트 기반 후속 처리, 장애 격리, 티켓 artifact 저장, 서비스 간 보안 통신, 관측성을 자연스럽게 포함하기 때문이다.

프로젝트 결과는 기능 구현량보다 "구조를 적용했을 때 어떤 운영 문제가 줄었는가"를 수치와 검증 증거로 판단한다.
